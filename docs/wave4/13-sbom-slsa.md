**Status:** Implemented. Active as of the next `api-v*` or `agent-v*`
tag push.

---

## 1. What is an SBOM?

A **Software Bill of Materials** (SBOM) is a machine-readable inventory
of every software component — libraries, OS packages, and their
transitive dependencies — that make up a released artifact.

For `cmtraceopen-api`, the SBOM lists every Rust crate (with version +
crate registry URL) and every OS package in the distroless base image.
For the agent MSI, it lists every Rust crate in `Cargo.lock`.

### Why it matters for enterprise deployments

| Concern | How SBOM helps |
| ------- | -------------- |
| **Vulnerability management** | When a new CVE drops (e.g. `openssl` RCE), your security team queries the SBOM to instantly know whether the deployed image is affected — without grepping source. |
| **License compliance** | Legal/procurement can audit open-source licenses before the image reaches production. |
| **Dependency auditing** | SOC 2 / FedRAMP / ISO 27001 controls increasingly require artifact provenance. An SBOM satisfies the "third-party component inventory" control. |

## 2. What is SLSA provenance?

**SLSA** (Supply-chain Levels for Software Artifacts, pronounced
"salsa") is a framework that describes how a software artifact was
built. A **provenance** document answers:

- *What source commit* was used?
- *Which CI workflow* ran the build?
- *Which GitHub-hosted runner* executed it?
- *What build inputs* (environment variables, parameters) were set?

This information is signed with a cryptographic key so that any
downstream consumer can verify it without trusting the build system
operator.

### Build Level 3

**SLSA Build Level 3** is the highest level achievable with GitHub
Actions today. It requires:

1. The provenance is generated by a **hardened build environment** (the
   `slsa-github-generator` reusable workflow) that is cryptographically
   isolated from the user's build steps — the provenance cannot be
   forged by a compromised build script.
2. The provenance is signed with an **ephemeral key** via **Sigstore**
   (no long-lived signing keys to rotate or leak).
3. The provenance is **non-forgeable**: only the SLSA generator
   workflow (running inside the same GitHub Actions OIDC trust domain)
   can produce a valid SLSA3 attestation for this repository.

## 3. What is generated per release

### api-server image (`api-v*` tags)

| Artifact | Storage | Description |
| -------- | ------- | ----------- |
| **Cosign signature** | GHCR OCI referrer | Keyless ECDSA signature over the image digest, chained to the GitHub Actions OIDC identity |
| **CycloneDX SBOM** | GHCR OCI referrer + Actions artifact | Full component inventory in CycloneDX 1.4 JSON format, generated by Syft |
| **SLSA provenance** | GHCR OCI referrer | DSSE-signed SLSA Build Level 3 attestation covering the image digest |

### agent MSI (`agent-v*` tags)

| Artifact | Storage | Description |
| -------- | ------- | ----------- |
| **Authenticode signature** | Embedded in MSI | Azure Trusted Signing (ATS) cert — verifiable with `signtool verify` |
| **CycloneDX SBOM** | GitHub Release asset | Rust dependency inventory from `cargo cyclonedx` |
| **SLSA provenance** | GitHub Release asset (`multiple.intoto.jsonl`) | DSSE-signed SLSA Build Level 3 attestation covering MSI + SBOM hashes |

## 4. Verifying the api-server image

All commands below require
[cosign](https://github.com/sigstore/cosign/releases) and optionally
[syft](https://github.com/anchore/syft/releases) installed locally.

### 4.1 Verify the cosign signature

```bash
IMAGE=ghcr.io/adamgell/cmtraceopen-api:0.1.0

cosign verify \
  --certificate-identity-regexp \
    "https://github.com/adamgell/cmtraceopen-web/.github/workflows/publish-api.yml" \
  --certificate-oidc-issuer \
    "https://token.actions.githubusercontent.com" \
  "$IMAGE"
```

A successful output ends with a JSON block containing `"verified": true`
and the GitHub Actions workflow identity. Any verification failure means
the image was either not signed by this pipeline or has been tampered
with.

### 4.2 Verify the SLSA provenance attestation

```bash
IMAGE=ghcr.io/adamgell/cmtraceopen-api:0.1.0

cosign verify-attestation \
  --type slsaprovenance \
  --certificate-identity-regexp \
    "https://github.com/slsa-framework/slsa-github-generator/.github/workflows/generator_container_slsa3.yml" \
  --certificate-oidc-issuer \
    "https://token.actions.githubusercontent.com" \
  "$IMAGE" \
| jq -r '.payload | @base64d | fromjson | .predicate'
```

The decoded predicate shows the exact `buildType`, the
`invocation.configSource` (the workflow run URL + git ref), and the
`materials` list (source commit SHA). Confirm that
`invocation.configSource.uri` matches
`git+https://github.com/adamgell/cmtraceopen-web@refs/tags/api-v*` and
that `buildConfig.steps` contains only the steps from `publish-api.yml`.

### 4.3 Inspect the SBOM

Retrieve the attached SBOM without pulling the full image:

```bash
IMAGE=ghcr.io/adamgell/cmtraceopen-api:0.1.0

# List attached SBOMs
cosign download sbom "$IMAGE"

# Or use syft directly (downloads and scans the registry image)
syft scan "$IMAGE" --output cyclonedx-json
```

`syft scan` should list every Rust crate from `Cargo.lock` plus the
distroless base image OS packages. Cross-reference against the
published `Cargo.lock` at the matching source tag to verify completeness.

### 4.4 Example: integrate into an Ansible deploy pre-flight

```yaml
# In your Ansible deploy role, before `docker compose pull`:
- name: Verify api-server image signature before deploy
  command: >
    cosign verify
    --certificate-identity-regexp
      "https://github.com/adamgell/cmtraceopen-web/.github/workflows/publish-api.yml"
    --certificate-oidc-issuer
      "https://token.actions.githubusercontent.com"
    ghcr.io/adamgell/cmtraceopen-api:{{ api_version }}
  changed_when: false
  register: cosign_verify_result
  failed_when: cosign_verify_result.rc != 0
```

A failed verification aborts the deploy — a poisoned image cannot
pass the OIDC-backed signature check.

## 5. Verifying the agent MSI

### 5.1 Verify Authenticode signature

```powershell
# Quick check (trusts Windows certificate store):
signtool.exe verify /pa /v /all CMTraceOpenAgent.msi

# Expected chain:
#   Microsoft Identity Verification Root Certificate Authority 2020
#     Microsoft ID Verified Code Signing PCA 2024
#       <Legal entity name>
```

### 5.2 Verify SLSA provenance

Install the [slsa-verifier](https://github.com/slsa-framework/slsa-verifier/releases):

```bash
# Download the MSI, SBOM, and provenance from the GitHub Release
RELEASE=https://github.com/adamgell/cmtraceopen-web/releases/download/agent-v1.0.0

curl -LO "$RELEASE/CMTraceOpenAgent.msi"
curl -LO "$RELEASE/sbom-agent.cyclonedx.json"
curl -LO "$RELEASE/multiple.intoto.jsonl"

# Verify provenance for the MSI
slsa-verifier verify-artifact CMTraceOpenAgent.msi \
  --provenance-path multiple.intoto.jsonl \
  --source-uri github.com/adamgell/cmtraceopen-web \
  --source-tag agent-v1.0.0

# Verify provenance for the SBOM
slsa-verifier verify-artifact sbom-agent.cyclonedx.json \
  --provenance-path multiple.intoto.jsonl \
  --source-uri github.com/adamgell/cmtraceopen-web \
  --source-tag agent-v1.0.0
```

A successful run prints `PASSED: Verified SLSA provenance`. Any failure
means the provenance document was either not generated by this pipeline
or the artifact hash no longer matches (tampering detected).

### 5.3 Inspect the SBOM

```bash
# View the CycloneDX SBOM
cat sbom-agent.cyclonedx.json | jq '.components[] | {name, version}'

# Or use syft
syft convert sbom-agent.cyclonedx.json --output table
```

## 6. CI workflow architecture

### publish-api.yml (api-server)

```
push api-v* tag
       │
       ▼
┌─────────────────────────────────────────────────────┐
│  build job                                          │
│  ─────────────────────────────────────────────────  │
│  1. docker buildx build --push (linux/amd64+arm64)  │
│     → outputs: digest = sha256:abc...               │
│  2. cosign sign (keyless OIDC)                      │
│  3. anchore/sbom-action → sbom.cyclonedx.json       │
│  4. cosign attach sbom                              │
└─────────────────────────────────────────────────────┘
       │ needs: build
       ▼
┌─────────────────────────────────────────────────────┐
│  provenance job (slsa-github-generator reusable)    │
│  ─────────────────────────────────────────────────  │
│  generator_container_slsa3.yml@v2.0.0               │
│  → reads Actions run metadata (actions: read)       │
│  → signs provenance with ephemeral Sigstore key     │
│  → pushes DSSE envelope to GHCR as OCI referrer     │
└─────────────────────────────────────────────────────┘
```

```

       ▼
```

## 7. Key management

Neither workflow uses a long-lived signing key.

| Workflow | Mechanism | Key lifetime |
| -------- | --------- | ------------ |
| api-server cosign sign | GitHub OIDC → Sigstore Fulcio CA | Ephemeral cert per run (<10 min TTL) |
| api-server SLSA | GitHub OIDC → Sigstore Fulcio CA | Ephemeral cert per run |
| agent MSI Authenticode | GitHub OIDC → Azure Trusted Signing HSM | Short-lived cert per signature (Microsoft-managed) |
| agent MSI SLSA | GitHub OIDC → Sigstore Fulcio CA | Ephemeral cert per run |

All signatures are recorded in the **Sigstore transparency log (Rekor)**,
providing an immutable audit trail even if the GitHub repository is
deleted.
